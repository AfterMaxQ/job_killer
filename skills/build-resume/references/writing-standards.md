# 简历撰写规范

> 供 build-resume SKILL.md 阶段三引用。Agent 在内容头脑风暴前 Read 此文件，遵循其中全部规范。

## 一目十行设计原则

**（HR 6-10 秒扫读）**

- **动词前置**：每个要点以动词开头（设计、实现、优化、搭建），不用「参与了」「负责」等模糊词
- **数字前置**：量化指标紧随动词后，阿拉伯数字
  - ✅ `测试集 AUC 0.98，KS 86%，模型区分能力优秀`
  - ❌ `通过大量实验最终取得了不错的效果`
- **技术落地**：不写「使用了 XX」，写「用 XX 解决了什么、达到什么效果」
- **密度优先**：删掉所有连接词、过渡句、客套话。每颗"信息子弹"：动作 + 技术 + 场景 + 结果
- **无括号说明**：技能名称不附加括号注解（如版本号、配置细节）。`JWT` 不写 `JWT（HS512）`，`Elasticsearch` 不写 `Elasticsearch（IK 分词）`。技术细节留给面试展开
- **技能精选**：不罗列重合度高的技术栈，单行不超过 5 个技术名词。只挑最关键最核心的，JD 明确要求的优先写。面试官看到 10 个技术堆在一起只会觉得你在凑数，3-5 个精准匹配的技术名词比 10+ 个泛泛而谈更有说服力
- **禁止随意将 API 调用冒充技能**：不得随意将平台/API 的基础使用方式列为技能，除非JD里确实特别要求了该为必须掌握的使用方式。这些是大模型 API 的基础调用模式，写成技能反而暴露对 AI 领域理解浅。同理适用于所有领域——普通调包不算技能
- **熟练度可选**：默认不加熟练度修饰词。用户明确要求时，使用「熟练」或「掌握」，禁止使用「精通」。熟练度加在类别名后，如「后端（熟练）」

## 项目经历撰写规范

### 标准结构（单项目通用）

```
项目名称 + 角色（独立开发/核心开发/模块负责人）| 时间（右对齐）
技术栈：核心框架 + 中间件 + 数据库 + 工具（精准关键词，不堆砌）
项目内容：1句话讲清项目定位与**解决的痛点**（不超过2行）。痛点必须具体——谁遇到什么问题、现有方案为什么不够好、你的方案怎么解决。背景不是功能介绍，是问题洞察。
核心工作：3 条，每条遵循「动作 + 技术 + 场景 + 结果」结构
项目成果：量化产出、优化效果、落地情况
```

### 写作原则

**0. 痛点驱动，先问"为什么做"**

项目内容的核心不是描述项目是什么，而是说清**为什么需要这个项目**。HR 看重的是你对问题的独特思考，不是功能罗列。

痛点写作三步法：
1. **谁有痛点** — 具体用户/场景（家庭用户、金融机构、校内学生）
2. **现有方案差在哪** — 竞品缺陷/行业空白/效率低下（"公有云盘隐私泄露、会员收费""信贷审批依赖人工、效率低"）
3. **你的方案怎么解决** — 一句话讲清核心思路

- ❌ `本项目是一个基于Spring Boot的云盘系统，支持文件上传下载` → 这是功能描述，没看到问题
- ✅ `面向家庭场景的私有云盘系统，解决公有云盘隐私泄露与大文件传输受限问题` → 有场景、有竞品缺陷、有解决方案

- ❌ `实现了信用风险预测模型，用于评估贷款违约概率` → 泛泛而谈
- ✅ `为金融机构构建自动化信用风险评估系统，替代人工审批效率低、标准不统一的痛点` → 知道为什么做、解决什么

**1. 动词前置，精准表述**

每条工作用技术动词开头，避免「参与了」「做了」「负责」等模糊表述。
常用动词：开发、设计、实现、优化、重构、封装、排查、解决、集成、搭建、部署、调优。

**2. 技术落地，不说空话**

不写「使用了 XX 技术」，写「用 XX 技术解决了什么问题、达到了什么效果（最好量化数据）」。
- ❌ `使用 Redis 做缓存`
- ✅ `引入 Redis 缓存高频数据，接口响应速度从 320ms 降至 45ms`

**3. 成果量化，4 个维度**

| 维度 | 示例 |
|------|------|
| 性能类 | 响应耗时从 320ms 降至 45ms；QPS 提升 6 倍 |
| 效率类 | 减少 70% 人工数据录入；自动化测试覆盖率 85% |
| 规模类 | 支撑 1200+ 用户；处理 10 万+ 条数据；开发 40+ 接口 |
| 成果类 | GitHub 80+ Star；竞赛省三等奖；上线零线上故障 |

**4. 技术亮点与作业拉开差距**

每个项目至少 1 个技术亮点：
- 基础优化：数据库索引优化、慢 SQL 调优、Redis 缓存、接口限流
- 架构设计：策略模式重构、通用工具封装、统一异常处理、分布式锁
- 工程化：Docker 容器化、CI/CD、单元测试覆盖、Swagger 接口文档
- 问题解决：排查内存泄漏、解决跨域/数据一致性、修复并发 Bug

### 正反示例

**❌ 反面（90% 学生写法）**
```
项目名称：电商系统
技术栈：SpringBoot、MySQL
负责后端开发，实现了商品和订单功能，完成了项目的开发任务。
```

**✅ 正面**
```
校园二手交易小程序后端 | 2025.03-2025.05 | 核心开发
技术栈：SpringBoot 3 + MyBatis-Plus + MySQL 8.0 + Redis + JWT
项目内容：面向本校学生的二手交易平台，支撑商品发布、在线沟通、订单管理

- 独立完成用户、商品、订单 3 个核心模块接口开发，封装统一返回结果与全局异常处理，输出 40+ RESTful 接口
- 针对商品列表查询慢问题，为 MySQL 建立联合索引，引入 Redis 缓存热点数据，接口响应从 320ms 降至 45ms
- 基于 JWT 实现登录认证与权限校验，集成微信登录，解决多端身份同步问题

项目成果：注册用户 1200+，上线后零故障运行，GitHub 80+ Star
```

### 分岗位侧重点

| 岗位 | 突出 | 弱化 |
|------|------|------|
| 后端开发 | 接口设计、数据库优化、中间件、并发、性能调优 | 前端页面、UI 实现 |
| 前端开发 | 组件封装、性能优化（首屏/打包）、兼容性、工程化 | 底层数据库、服务端逻辑 |
| 算法/数据挖掘 | 数据清洗、特征工程、模型选型调优、指标提升 | 业务 CRUD、页面开发 |
| 软件测试 | 测试用例、缺陷管理、自动化脚本、覆盖率 | 功能开发细节 |

### 高频避坑

| 错误 | 正确 |
|------|------|
| 项目介绍长篇大论，个人贡献一笔带过 | 项目内容 1 句话，核心篇幅留给具体工作 |
| 堆砌技术名词和实际工作无关 | 只列实际使用的技术，面试会追问 |
| 团队项目全算自己头上 | 诚实写明负责的模块 |
| 课程作业千篇一律无亮点 | 加 Redis 缓存、JWT 认证、Docker 部署升级为亮点项目 |
| 实习生写"精通 Java" | 写"熟练使用""掌握"即可 |

## 教育背景撰写规范

### 核心原则

**门槛优先**：学校、学历、专业、毕业时间放在最显眼位置，帮助 HR 3 秒判断学历是否达标。
**匹配为王**：所有补充信息（GPA、课程、奖项）围绕投递岗位展开，无关内容果断舍弃。
**扬长避短**：优势项（高绩点、排名）前置放大，劣势项（低绩点、院校普通）弱化处理，绝不造假。
**真实可查**：所有数据如实填写，大厂背调会核对教务系统。

### 标准结构（单段学历）

```
【学校全称】 【专业全称】 【学历】 | 【入学时间-毕业时间】
- GPA/专业排名：有优势则写，标注满分/总人数
- 核心课程：8-10门，与岗位高度匹配
- 学业荣誉：奖学金等学业类奖项，按含金量排序
```

**排版位置规则：**
- 常规：放在「基本信息」之后、「专业技能/项目经历」之前，HR 先过学历门槛再看能力
- 特殊：院校背景普通但项目/竞赛突出，可移至「项目经历」之后，先秀硬实力降低学历短板影响

### 分模块撰写技巧

#### 1. 基础信息：精准无歧义

- **倒序排列学历**：本科只写大学，研究生先写研究生再本科，不写高中
- **专业名称规范**：按学籍/毕业证名称书写，如「软件工程」不写成「计算机软件开发」；有辅修/双学位可标注
- **毕业时间精确到月**：如「2023.09-2027.06」，方便 HR 判断可实习时长与转正潜力
- **特殊学历如实标注**：非全日制、专升本等坦诚说明

#### 2. GPA 与排名：量化呈现，扬长避短

**✅ 推荐写（满足其一即可）：**
- GPA ≥ 3.5/4.0（或均分 85/100 以上），或专业排名前 20%
- 专业课 GPA 远高于整体 GPA：只写专业课 GPA，标注「核心专业课 GPA 3.8/4.0」
- 绩点呈明显上升趋势：如「大三学年 GPA 3.8/4.0，较大一提升 0.6」

**标准写法：**
- 写法 1（GPA + 排名）：`GPA 3.72/4.0，专业排名 11/136（前8%）`
- 写法 2（只写排名）：`连续2学年专业排名前12%`
- 必须标注满分（4.0/5.0），避免 HR 误判；排名优先于 GPA

**❌ 不推荐写：**
- GPA 低于 3.0/4.0（或均分 75 以下），且无突出排名
- 大一、大二学生修课较少，绩点参考价值低

#### 3. 核心课程：精准匹配，拒绝堆砌

**筛选原则：**
- **岗位优先**：只列和投递岗位直接相关的课程，JD 关键词对应课程放最前面
- **专业课为主**：只写专业核心课、方向选修课，不放公共课（马原、大学英语、高等数学、体育）
- **控制数量**：8-10 门最佳，最多不超过 12 门
- **高分加分**：核心课程分数 90+ 可标注，如「数据库系统原理（94分）」

**分岗位参考（计算机类）：**

| 投递岗位 | 核心课程推荐 |
|----------|------------|
| Java 后端开发 | 数据结构与算法、计算机网络、操作系统、数据库系统原理、Java程序设计、软件工程、分布式系统原理 |
| 前端开发 | 数据结构、计算机网络、Web前端技术、JavaScript程序设计、移动应用开发、软件工程 |
| 算法/数据挖掘 | 数据结构、概率论与数理统计、线性代数、机器学习、深度学习、数据挖掘、自然语言处理 |
| 软件测试 | 软件工程、软件测试技术、数据库原理、计算机网络、Python程序设计 |

**排版优化：** 可按类别拆分：
- 核心课程：计算机网络、操作系统、数据结构与算法、数据库系统原理
- 方向课程：Java企业级开发、分布式系统、微服务架构、软件工程

#### 4. 学业荣誉：只放高相关、高含金量

- 教育背景下只放学业类奖项，技术竞赛单独放「竞赛获奖」模块
- 排序：国家级 > 省级 > 校级 > 院级
- 写法：奖项名称 + 获奖级别 + 难度/比例，如「校级一等奖学金（专业前5%，连续2学年）」
- 歌唱比赛、运动会等与技术实习无关的内容全部删掉

### 分场景差异化策略

#### 1. 名校/高绩点选手：放大优势

教育背景前置，重点突出学校 title、GPA、专业排名、国家级奖学金、保研资格；可补充实验室经历、科研项目、导师研究方向。

#### 2. 双非/绩点普通选手：弱化短板，突出匹配

- 缩短教育背景篇幅，去掉 GPA 和排名（GPA < 3.0 时）
- **重点写匹配岗位的核心课程**，标注高分专业课，体现专业基础达标
- 项目经历、竞赛获奖可移到教育背景之前，先秀硬实力
- 补充权威在线学习认证（Coursera 专项证书、官方技术认证），弥补院校不足

#### 3. 跨专业求职选手：弥补专业缺口

- 明确标注本专业 + 辅修/双学位（如有）
- 核心课程列计算机相关的自学课程、辅修课程、在线认证课程
- 补充「在线学习」子模块，列出系统学习过的技术课程与认证
- 重点放在项目、技能模块，用实战经历证明能力

#### 4. 研究生选手：突出科研与方向

先写研究生再写本科；研究生阶段重点写研究方向、导师、科研项目、论文、专利；本科阶段简化处理。

## 自我评价撰写原则

**撰写原则：紧扣岗位类型，不写无关卖点。**

| 岗位类型 | 突出 | 避免 |
|----------|------|------|
| 技术岗（后端/AI/算法） | 技术热情、执行闭环、学习能力、技术栈体系 | 商务英语、翻译、跨文化沟通（与编码无关） |
| 产品/运营岗 | 用户洞察、数据分析、沟通协调、业务理解 | 底层技术细节 |

**反面示例（投技术岗，却写商务英语 — 卖点错位）：**
```latex
{1. ...对 AI 在金融场景的落地有浓厚兴趣...}
{2. 英语 CET-6，具备全英学术写作与商务沟通能力；自主学习能力强...}
```
问题：第 2 点以前置「商务英语」开场，技术岗 HR 不关心商务沟通能力。英语能力可放在教育背景中简要提及，不应占自我评价的独立一点。

**正面示例（技术岗）：**
```latex
{1. 软件工程 + 经济学复合背景，对 AI 技术保持高度热情，执行能力强、主动闭环任务，可快速适应新环境}
{2. 自主学习能力突出，系统掌握 Java 后端与机器学习技术体系，热衷探索 AI 在业务场景中的创新应用}
```

---

> **大道至简**：不要加上连续的无逻辑修饰词（例如：软件工程 + 经济学 + 数学统计建模复合背景）显得在强行给自己加头衔！
