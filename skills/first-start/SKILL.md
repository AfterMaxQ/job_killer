# First Start — 首次使用引导

## 概述

帮首次使用者理解工具链 + 将整个技能生态适配到你的求职领域。不直接产出简历/话术，是让其他功能在你手里发挥最大效果的"新手村"。

## 触发条件

- 用户说 "first_start"、"开始"、"帮我"、"引导"、"怎么用"、"新用户"
- 项目首次打开，用户像在探索而非执行具体任务时
- Agent 判断当前项目状态后主动建议

## 核心流程

### Step 0：检测项目状态

在展示选项之前，先检查：

1. `个人信息/` 是否存在？内容是否完整？
2. 各 skill 是否已被领域适配过？（检查 interview-question-generator 的分类表是否非 CS 领域维度、greeting-generator 的示范是否非 CS 岗位）

**分支逻辑：**

| 状态 | 行为 |
|------|------|
| `个人信息/` 不存在或不完整 + Skill 未适配 | 正常走 A/B 选项 |
| `个人信息/` 完整 + Skill 已适配 | 提示："之前已经适配到 [方向]，你想——A. 重新了解功能 B. 换方向重新适配 C. 更新个人信息" |
| `个人信息/` 完整 + Skill 未适配 | 提示："你的个人信息已经有了，但系统还没适配到你的领域。建议先做 B 选项" |

### Step 1：欢迎 + 两个选项

> 你好！我是你的求职助手。你现在可以——
>
> **A. 帮我介绍一下这些功能** — 了解每个技能做什么、什么时候用
> **B. 帮我初始化系统** — 告诉我你的求职方向，我把整个系统适配到你的领域，然后帮你建档案

只问这一次，等用户选择。用户直接打字回复 "A" 或 "B" 即可。

---

## 选项 A：功能介绍

用户选 A 后，简洁地介绍整个技能生态。

### A1：一句话定位

| 技能 | 一句话 |
|------|--------|
| `init-personal-info` | 建档案——把你的个人信息结构化存起来，后续所有功能都从这里读 |
| `build-resume` | 做简历——粘贴 JD，自动生成针对这份 JD 的定制简历（Markdown + PDF） |
| `greeting-generator` | 写招呼语——Boss/猎聘上主动触达 HR 的第一条消息，前 18 字决定 HR 点不点开 |
| `hr-communicator` | 回消息——HR 回复你之后，帮你出沟通话术（约面试/谈薪资/问离职原因等 8 类场景） |
| `interview-question-generator` | 备面试——根据 JD + 你的项目，生成口语化面试 Q&A，教你讲出设计思路 |

### A2：什么时候用哪个

| 我遇到了... | 用这个 |
|------------|--------|
| 刚拿到一个 JD，想投这个岗位 | `build-resume` |
| 在 Boss 直聘看到一个好岗位，不知道怎么开口 | `greeting-generator` |
| HR 回我了，问期望薪资/为什么离职/什么时候面试 | `hr-communicator` |
| 收到面试通知，不知道怎么准备 | `interview-question-generator` |
| 换了个求职方向/首次使用，建档案让 AI 记住你 | `init-personal-info` |

### A3：回到选择

> 还需要做 B 选项——初始化适配你的领域吗？（回复"B"或"不用"）

---

## 选项 B：领域适配 + 初始化

这是 first-start 的核心——将 5 个 skill 深度适配到用户的求职领域，然后自动初始化个人信息。

### B1：收集领域信息（精简 3 问）

**一次只问一个**，用户回答后再问下一个。

**第 1 问：行业/领域**
> 你找什么方向的工作？（如建筑设计、金融分析、运营市场、医疗卫生...）

**第 2 问：目标岗位**
> 具体是什么岗位？（可多个，如建筑设计师/方案设计师）

**第 3 问：当前身份**
> 目前是应届生、在职跳槽、还是转行？

### B2：推断领域特征 + 确认

基于 B1 的 3 个回答，推断该领域的以下特征，一次性呈现给用户确认。

**推断内容格式：**

```
根据你说的 [行业] — [岗位]，我推断这个领域的招聘特征大概是这样的：

🔹 能力维度（JD 通常看重的）：
   [推断维度 1]、[推断维度 2]、[推断维度 3]、[推断维度 4]...

🔹 面试方向（面试通常问什么）：
   [方向 1]、[方向 2]、[方向 3]...

🔹 项目/作品类型：
   [类型 1]、[类型 2]、[类型 3]...

这些推断准吗？需要调整的地方告诉我~
```

**推断原则：**
- 基于用户回答 + Agent 对该领域的常识推断
- 不确定的维度标注"（？）"让用户修正
- 如果 Agent 对这个领域确实不熟，坦诚说"我对 [领域] 了解有限，你帮我补充几个关键的能力维度"

### B3：改写 5 个 skill

用户确认 B2 的推断后，Agent 逐个改写 5 个 skill 文件的示例和分类体系。

**改写铁律（不可违反）：**
1. **只改示例和分类维度**，不动流程结构、Phase 数量、输出格式
2. **保留核心哲学**：build-resume 的 STAR 原则、greeting-generator 的前18字规则、hr-communicator 的 4-step、interview-question-generator 的口语化回答结构——全部不动
3. **推断的维度必须已经过用户确认**，不新增未确认的内容
4. **不确定时坦诚**，让用户补充

## 改写原则总结

1. **只改示例和分类维度**，不动流程结构、Phase 数量、输出格式
2. **保留核心哲学**：build-resume 的 STAR 原则 + greeting-generator 的前18字规则 + hr-communicator 的 4-step + interview-question-generator 的口语化回答结构——全部不动
3. **推断后必须用户确认**，不编造不熟悉的领域细节
4. **不确定时坦诚**，让用户补充——"我对 [领域] 了解有限，你帮我调整一下"
5. **用户之后可以手动再改**——改写后告诉用户"如果需要调整某个 skill 的示例或维度，随时告诉我，随时改"

## 常见错误

| 错误 | 正确做法 |
|------|----------|
| 用户选了 A，介绍完后忘记问是否需要 B | A3 强制回到选择界面 |
| B2 跳过确认直接改写 | 推断后必须等用户确认或修正 |
| 改写 interview-question-generator 时动了 6 个 Phase 的流程 | 只改 Phase 2/4/5 的分类维度和 Phase 6 的示例，Phase 0/1/3 流程不变 |
| 对不熟悉的领域硬编维度 | 坦诚说不知道，让用户补充 |
| B4 告诉用户"现在可以输入 /init-personal-info" | 直接自动启动 init-personal-info，不让用户额外操作 |
| 改写 skill 时保留 CS 残余 | 改写后检查——如果用户做建筑设计，interview-question-generator 里不应再出现"后端""前端""缓存"等词 |
| 初次使用已经 init 过的用户，不检查状态直接走全套 | Step 0 先检查，根据状态分支处理 |
